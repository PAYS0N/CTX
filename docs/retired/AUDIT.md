Audit report

  The code itself is in good shape — small files, clean seams, honest error taxonomies, hermetic tests. The problems are almost all in the context layer: the material CTX injects into a
  low-reasoning model's window is currently the least trustworthy part of the system, and several documented contracts don't match what the tools actually do. For a Sonnet-low-effort
  consumer, that's exactly backwards — a weak model amplifies whatever it's fed and can't detect contradictions.

  Findings, ranked by impact on the stated goal.

  Critical: the always-injected context is wrong or contradictory

  1. Root intent.md teaches the retired architecture on every session. .context/intent.md (injected on essentially every read) still says "Layer 2 — ctx-access is the sanctioned path to
  source (chain read root→target, write-needs-read, task lifecycle)". That contradicts CLAUDE.md's "use native Read/Edit" in the same context window. A low-effort model resolving that
  conflict is a coin flip — some sessions will go hunting for a ctx-access binary. Note intent.md is owner-authored and never regenerated by ctx-scan, so the pending reseed will not fix
  this; it needs a manual edit now.

  2. README.md:27-36 documents the deleted ctx-access init-task/end-task quickstart as the workflow. Same failure mode, and it's the first file anything reads.

  3. The CLAUDE.md ↔ ctx-verify contract is factually wrong. Both CLAUDE.md files say done = it prints {"status":"pass"}. I ran ctx-verify ctx-core: the default terse mode prints pass;
  {"status":...} only exists behind --json (crates/ctx-verify/src/cli.rs:71-73). A literal-minded model looking for the documented string will conclude the checkpoint failed or start
  debugging. Fix one side — either document pass, or make the JSON envelope the default.

  4. Stale .ctx content is served with zero staleness signal. ctx-context never consults hashes.json, even though staleness is locally computable from data already on disk. Right now the
  entire ctx-cage subtree, docs/, and others are stale (ctx-scan --check confirms), and rollups confidently describe deleted systems (the crates/ rollup describes a ctx-check crate that
  doesn't exist; the ctx-summarize rollup describes the dead FromCache/task-cache flow). The single highest-leverage code change in this audit: have serve.rs cross-check the sidecar and tag
  sections [rollup — STALE] or add a one-line banner. A weak model can act on "this summary may be outdated"; it cannot act on silence.

  5. The "sealed spec + superseding amendment" pattern is a reasoning tax the target model can't pay. docs/SPEC.md requires readers to apply the rev-4 paragraph over body prose that still
  specifies ctx-access, task lifecycles, and stale banners. The docs rollup even instructs "read the amendment before the section it amends." Sonnet-low-effort will read the body literally.
  Since you're open to restructuring: rewrite the body to rev-4 reality and keep superseded text in git history / ADRs, which is what ADRs are for. Same for SANDBOX.md — a retired-banner
  file is still a file a grep can land in; consider moving retired designs to docs/retired/.

  High: retired architecture still live in code and help text

  6. Dead ctx-access machinery ships in ctx-summarize. cache.rs and the FromCache CLI mode (cli.rs:36) read .context/.cache/<task-id>.json, which nothing writes anymore (ctx-context writes
  hook-<session>.json). It's dead functionality presented as a supported mode.

  7. User-facing help text lies: crates/ctx-cage/src/cli.rs:22 — about = "Generic sandboxed-agent harness: brokered ctx-access/ctx-verify over a project." — and :76 says self-test "runs
  ctx-access manifest". --help output is prime context for an agent. Five crates still reference ctx-access in doc comments (grep -rn ctx-access crates/); those comments become leaf .ctx
  content on the next reseed, re-poisoning the fresh tree.

  High: the summary generator has no quality gate

  8. Line budgets are unenforceable as designed. prompts/summarizer-rollup.md:54 says "the line budget is fixed in docs/SPEC.md" — but the generating model never receives SPEC.md, so the one
  place the budget is stated is the one place the generator can't see. The docs rollup's claim that SPEC's numbers "must stay numerically identical to the prompt" is already broken: the
  prompt contains no numbers. Put the numbers in the prompts.

  9. No mechanical validation of generated summaries. Nothing checks output length, the banned filler phrases, hedging (the committed .claude/settings.json.ctx says "presumably
  records/updates context state" — a hedge the leaf prompt bans in spirit, now served as fact), or the intent_divergence: label format. Since garbage in a leaf propagates upward through
  rollups, a cheap post-generation lint in ctx-scan (line count, banned-phrase grep, retry-once-on-fail) would harden the whole chain. The adversarial-reread rule currently relies on the
  summarizer model obeying prose — the same trust-the-model approach the rest of CTX explicitly rejects.

  10. The adapter can silently write a truncated summary. agents/summarizer-claude.py never checks stop_reason; at max_tokens 2048 a large rollup gets cut mid-sentence and written as truth.
  One if response.get("stop_reason") == "max_tokens": die(...) fixes it. Also CTX_AGENT_TEMPERATURE is documented in the docstring but never read — dead option.

  11. Freshness ≠ integrity. The hash tree (crates/ctx-scan/src/hash.rs) only hashes sources. A deleted, hand-edited, or never-generated .ctx/rollup.ctx is invisible to --check — the chain
  silently serves (absent: …) forever, and assemble_rollup_input (crates/ctx-summarize/src/runner.rs:116-119) silently omits any subtree lacking a rollup.ctx, so a parent rollup can
  confidently describe a directory minus entire children. --check should also verify expected artifacts exist.

  Medium

  12. Hook coverage gaps (.claude/settings.json): the matcher is Read|Grep|Glob, but hook.rs was built for Edit too (it parses file_path from Edit events, and its doc comment says
  "Read/Edit"). An agent that Writes a new file or Edits a file it never read gets no chain. Adding Edit|Write to the matcher is free — dedup already prevents re-injection noise. Separately,
  once-per-session dedup interacts badly with context compaction in long sessions: the chain injected at turn 3 may be summarized away by turn 80 with no way to get it back except manual
  ctx-context. Consider a re-inject-on-Edit or making that failure mode explicit in CLAUDE.md ("if you've lost the chain, run ctx-context <path>").

  13. ctx-verify "pass" can mean "didn't run". ToolMissing → Skipped, and skipped never affects status (crates/ctx-verify/src/checks.rs:190, model rollup confirms). If python3 or
  cargo-machete's wrapper is missing, the gate shrinks silently and still prints pass. The terse render should at least name skipped checks (pass (skipped: rationale, machete)).

  14. Scoped runs aren't scoped for doc/test. effective_args inserts -p <pkg> but leaves --workspace in the doc and test specs (checks.rs:100-115), so cargo unions them and ctx-verify
  <crate> still docs/tests the whole workspace — the promised tight loop isn't. Strip --workspace when a package is given.

  15. Template/root mirroring is honor-system. Doctrine says the lint configs must mirror; clippy.toml is byte-identical but the Cargo.toml lint tables differ (comments only, today), the
  root rollup's "byte-identical" claim is already false, and no script compares them. Add a sync check to the ctx-verify battery — this is exactly the kind of drift the repo's philosophy
  says to make mechanical.

  16. A // rationale: placement trap: has_rationale_before (scripts/rationale_check.py:120-132) skips blanks and attributes but not doc comments, so // rationale: placed above a /// block
  silently doesn't count. A weak model will loop on "I added the rationale, why does it still fail?" Either skip /// lines in the scanner or state the placement rule in CLAUDE.md.

  Minor

  - .claude/settings.local.json (untracked, user-local, currently in your git status) is in summarization scope — it'll churn hashes per user and get billed summaries. Add it to .ctxignore.
  - CLAUDE.md grammar/typos ("a directories contents", trailing spaces) — worth fixing only because it's the highest-read file per token in the repo.
  - The leaf-prompt example (prompts/summarizer-leaf.md:69-87) is realistic enough that its own .ctx warns it could be mistaken for a real auth module; a one-line "example — fictional file"
  header inside the example would defuse it for weak generators.
  - Mixed .ctx formats in the committed tree (old file:/behavior: YAML-ish vs. new prose) — resolved by the pending reseed, but reseed after fixing items 6–8, or the fresh tree inherits the
  poison.

Root cause analysis:

Root-cause analysis by your categories, plus two classes I think sit underneath them.

  a. Context issues (1, 2, 3, 5) — manual docs drift because nothing owns them

  Cause: The system has a freshness pipeline for derived context (hash tree → regenerate .ctx) but the authored layer — intent.md, README, CLAUDE.md, SPEC — sits entirely outside it. Worse,
  those docs restate facts that are derivable: intent.md restated SPEC's architecture, README restated the workflow, CLAUDE.md restated ctx-verify's output contract. Every restated fact is a
  copy with no sync mechanism, and copies drift. The repo applied "make bad states unrepresentable" to Rust and exempted prose.

  Fixes, in order of leverage:

  1. Make the tools self-describing and assemble the doc from them. The reason CLAUDE.md lied about {"status":"pass"} is that the contract lives in two places: cli.rs and prose. Give each
  binary a --contract (or reuse --help) that emits its one-paragraph agent-facing contract, and generate the tool section of CLAUDE.md/README from those at build time (a ctx-verify battery
  item asserts the committed doc matches the assembled output). Then a contract change that doesn't update the doc fails the checkpoint. Single source becomes the binary, which is the thing
  that's actually true.
  2. Split intent from description, and audit the seam you already built. intent.md went stale because it contained architecture ("ctx-access is the sanctioned path"), which is description,
  not intent. Rule: intent.md holds only goals, non-goals, and invariants — things that survive a pivot; anything describing current mechanism belongs in the regenerated rollup. And this is
  exactly what Layer 3's auditor was designed to catch (intent vs. rollup divergence) — it exists as a prompt and is deferred. Wiring auditor.md into ctx-scan --update (audit each
  regenerated rollup against its intent, report divergence) would have flagged finding #1 mechanically. The fix for "manual upkeep" isn't more upkeep discipline; it's activating the audit
  layer you already specced.
  3. Separate the governance ledger from the consumption artifact (fixes 5). SPEC's sealed-body-plus-amendment pattern conflates two documents: an append-only record of what was decided
  (that's DECISIONS.md's job, and git's) and a currently-true spec an agent can read literally. Keep the seal as a process rule (changes require an ADR), but make each re-spec rewrite the
  body. The consumption artifact must be internally consistent at every commit; history belongs in the ledger. Same logic retires SANDBOX.md to docs/retired/ — a weak model's grep shouldn't
  be able to land in a retired design presented at the same path-prestige as live ones.
  4. The deeper point in your "built for per-file context" observation: the mirror tree gives every file an owner and a freshness bit, but cross-cutting truths (which era are we in, what's
  the workflow) have no node — so they got stuffed into hand-maintained prose. The root rollup.ctx is actually the natural home for "current shape": it's regenerated, it sees every child,
  and it's injected first. If STATUS.md's "current shape" section were trimmed to pointers and the root rollup were trusted as the orientation document, the per-file machinery would cover
  the central-doc role too. What can't be derived (goals, decisions) stays small enough that manual upkeep is rare rather than constant.

  b. Missing features (4, 9) — trust boundary drawn in only one direction

  Cause: The system distrusts the coding agent (lints, gates, deny-lists) but trusts its own outputs: summaries are served unverified and unmarked, generator output is written to disk
  unchecked. Nobody owned the consumer-side question "how does the reader know this is current and well-formed?" — hashes.json was designed for the regeneration path, and the serving path
  never got a stakeholder. Also, the pipeline was validated on fresh fixtures; the degraded state (stale tree, truncated output) was never a test case, so its behavior defaulted to "serve

  c. Small correctness errors (6, 7, 8, 14, 16) — three distinct causes

  6, 7 (dead ctx-access code/help text): incomplete pivot, no retirement mechanism. ADR-035/036 deleted the concept but nothing enforced the deletion. Fix in the repo's own idiom: a retired_terms_check.sh in the ctx-verify battery — a list of banned
  identifiers (ctx-access, init-task, …) appended when an ADR retires something, allowed only in docs/DECISIONS.md and docs/retired/. It's the no_allow_check pattern applied to vocabulary, and it would have caught the code, the help text, the doc
  comments, and README/intent.md in one grep. Systemic rule: an ADR that retires something ships with its own grep.

  8 (budget invisible to generator): duplicated constant with the wrong copy authoritative. SPEC held the number; the prompt — the only artifact the generator sees — held a pointer. Rule: the consumer-visible copy is the single source; put budgets in the
  prompts, have SPEC point at them, and let the (b) write-time validator enforce the same number, which makes drift self-detecting.

  14 (-p + --workspace): hermetic tests can't see external-tool semantics. FakeRunner verifies "we passed -p" but nothing verifies cargo's interpretation. The fake-everything discipline is right, but it needs a complement: one thin boundary-contract test
  per wrapped tool (real cargo, trivial fixture, asserts the flag combination behaves as assumed). The convention "every crate fakes its boundary" should read "…and pins the real boundary's contract once."

  16 (rationale/doc-comment trap): heuristic gate whose false-failures aren't in its error message. Agents debug gates from the failure message alone — they won't read the script. Rule for every gate script: the error message is the documentation; it
  must state the exact accepted placement, including the interactions the heuristic has (doc comments, attributes). Plus the two-line fix of skipping /// lines.

  d. Gaps (10, 11, 12, 13, 15) — boundary failure modes were never enumerated

  Cause: All five live at a boundary with a system CTX doesn't control — the API (truncation), the human (hand-edited .ctx), the harness (compaction, matcher coverage), the toolchain (missing tools), the template consumer (drift). Internal invariants are
  documented and tested exhaustively; external misbehavior was mostly not asked about. And where it was asked (fail-open hook), the answer was applied inconsistently: the hook fails open by explicit ADR, but ctx-verify — a gate — also effectively fails
  open via Skipped, with no ADR saying so.

  Fixes:

  1. Adopt the crisp rule the repo almost states: serving fails open; gates fail closed. Under it, ToolMissing in the default battery becomes errored (or at minimum pass (skipped: rationale, machete) in the terse render — never a bare pass), and the hook
  stays as-is. One sentence in intent.md, enforceable in review.
  2. Make "must stay in sync" claims illegal without an enforcer (15, same disease as 8). A doc-lint that greps docs for sync-claims ("must stay in sync", "byte-identical", "mirror") and requires each to name its checking script would convert every such
  promise into either a check or an honest "should".
  3. For each external boundary, record the misbehavior contract as a short ADR checklist question ("what happens when the other side truncates / edits / forgets / lacks the tool?") — 10, 11, and 12 each fall out of one line of answer (check stop_reason;
  verify artifact existence in --check; add Edit|Write to the matcher and a CLAUDE.md line for post-compaction recovery).

  Two classes underneath yours

  e. Dogfooding asymmetry. Rust gets the full regime; prompts, the Python adapter, shell scripts, help text, and docs get nothing — and nearly every finding above lives in that unregulated zone (a: docs, b/8/9/10: prompts+adapter, 7: help text, 16:
  script). "Outside the lint regime" silently became "outside all regimes." The fix is one move covering many findings: a meta-battery in ctx-verify — doc-claims check, retired-terms check, prompt/budget consistency, adapter contract test (stub API,
  asserts truncation handling). The system's thesis is that mechanical gates beat discipline; apply the thesis to its own non-Rust half.

  f. Cost-gated validation. The stale tree, the mixed formats, and the unproven billed path all trace to "the real run is billed, so defer it." But the stub agent already exists and makes generation free — the reseed and the e2e fixture could run on stub
  continuously (validating the pipeline), reserving billed runs for a small periodic canary (validating the model output). Cause: billing was treated as gating the whole test instead of just the model call; the seam to split them is already built.

  If you want a single organizing principle out of all of this: every "must" in this repo should be either generated from its source of truth or checked by the battery — a "must" enforced by prose is the bug class you just experienced, five ways. The
  encouraging part is that none of the fixes require new architecture; they're the existing patterns (grep gates, hash tree, auditor prompt, stub agent, battery) applied to the parts of the system that were exempt.
